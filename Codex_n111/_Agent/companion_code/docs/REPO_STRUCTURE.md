# Repo Structure

## Назначение

Этот документ фиксирует,
как именно устроена папка `companion_code`
и где в ней должен лежать каждый тип артефакта.

---

## 1. Главный принцип

Папка `companion_code` должна быть repo-ready структурой.

То есть:

- исполняемый код не размазывается по корню;
- формальные доказательства лежат отдельно;
- notes не смешиваются со скриптами;
- outputs не лежат рядом с исходниками;
- корень содержит только общую навигацию и служебные файлы.

---

## 2. Layout

### `scripts/`

Главный кодовый слой.

Сюда входят:

- агрегирующие движки;
- layer-report скрипты;
- screening- и audit-инструменты;
- chemistry-карта кода.

Если появляется новый рабочий Python-скрипт,
его базовое место по умолчанию — именно `scripts/`.

### `formal_proofs/`

Формальные Lean-доказательства.

Сюда не нужно класть Python,
markdown-notes или outputs.

### `notes/`

Небольшие текстовые материалы,
которые сопровождают код,
но не являются корневой документацией репозитория.

### `outputs/`

Сохраняемые результаты запусков:

- полные текстовые отчёты;
- экспортированные таблицы;
- любые generated artifacts,
  если они нужны как reference-output.

Внутри `outputs/` допускаются тематические подпапки.

Например:

- `outputs/readable_reports/` — сохранённые человекочитаемые каталоги,
  построенные из report-скриптов.

### `docs/`

Служебная документация именно про структуру кода и навигацию по репозиторию.

---

## 3. Источник истины

На текущем этапе плоская папка
`/Users/abc/Codex_n111/_Agent/Code`
используется как source-snapshot
актуальных финальных Python-скриптов.

Но нормальная repo-форма —
это именно:

`/Users/abc/Codex_n111/_Agent/companion_code`

Следовательно:

- новые версии скриптов могут сначала появляться в `Code`;
- затем они синхронизируются в `companion_code/scripts`;
- после синхронизации именно `companion_code`
  считается правильной сопровождаемой структурой.

---

## 4. Что не надо делать

- не класть Python-скрипты в корень репозитория;
- не смешивать Lean и notes;
- не хранить outputs рядом с исходниками;
- не использовать `__pycache__` как часть репо-структуры;
- не использовать `.DS_Store` как meaningful artifact.

---

## 5. Текущее состояние

После синхронизации:

- `scripts/` должен совпадать по актуальному коду с `Code`;
- `formal_proofs/` должен сохранять Lean-слой;
- `notes/` должен сохранять research-notes;
- `outputs/` должен содержать сохранённые результаты;
- `outputs/readable_reports/` должен содержать сохранённые reference-catalogs
  для частиц и элементов;
- в корне должны быть:
  - `README.md`
  - `README_code.md`
  - `requirements.txt`
  - `.gitignore`
